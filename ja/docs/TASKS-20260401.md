## フェーズ 1 — インフラと弱いモジュール（2026 年 4 月）

**対象マイルストーン: M1**

### 柱 4: 新規ユーザーオンボーディング

- [ ] **T-001** — `scripts/quickstart.sh` を作成
  - `~/.claude/` ディレクトリを検出（無い場合はユーザー確認のうえ作成）
  - 最初のスラッシュコマンド＋CLAUDE.md＋スキルをユーザーのセットアップにコピー
  - CLAUDE.md にエージェント発見用の行を追記し、AGENT-INDEX.md にリンク
  - 冪等: 既存ファイルは警告を出してスキップし、上書きしない
  - 完了時に次のステップを表示する

- [ ] **T-002** — `QUICKSTART.md`（最初の 15 分ビジュアルガイド）を作成
  - 注釈付きターミナル手順をコードブロックとして記載（可能な限り PNG スクリーンショットを避け、ASCII か Mermaid を優先）
  - 各ビジュアルを生成するコマンドを記録し、後から再撮影できるようにする

- [ ] **T-003** — 全 10 モジュールの README に難易度バッジを追加
  - 形式: 各モジュール README の冒頭に `> 🟢 **Beginner** | ⏱ 30 minutes`
  - バッジの下に「What you'll build」の箇条書きプレビューを追加

### 柱 7: 生きたカリキュラム

- [ ] **T-004** — `WHATS-NEW.md` を作成
  - 形式: `## CC vX.X — DATE` に機能変更を箇条書き
  - 現行 Claude Code バージョンのエントリを少なくとも 1 件追加

- [ ] **T-005** — 全 10 モジュールの README にバージョンバッジを追加
  - 形式: `> ✅ Verified against Claude Code **vX.X.X** · Last verified: YYYY-MM-DD`
  - 各モジュールのフロントマターに `cc_version_verified` を追加

- [ ] **T-006** — `.github/workflows/staleness-check.yml` を作成
  - スケジュール: 毎週月曜 09:00 UTC
  - 各モジュールのフロントマターから `last_verified` 日付を読む（`yq` または 10 行の Python スクリプトを使う）
  - 30 日以上検証されていないモジュールに対して GitHub Issue を起票
  - 同じタイトルの Open Issue が既にあればスキップ

### 柱 6: コンテンツ品質（CLAUDE.md）

- [ ] **T-007** — プロジェクトのルート `CLAUDE.md` を作成
  - プロジェクト自身の設定としてベストプラクティスを示す
  - 投稿規約、モジュール構造、メンテナンスワークフローを明文化

### 柱 2: AI エージェントインデックス（ジェネレータ）

- [ ] **T-008** — `scripts/build-agent-index.py` を作成
  - 全 10 モジュールの README ファイルを読む
  - 抽出: 見出し（→ トピック）、コードブロック（→ 言語タグ付きの例）、テーブル（→ 参照データ）
  - CEO プランで定義したスキーマで `agent-manifest.json` を出力
  - `AGENT-INDEX.md`（人間可読サマリ）を出力
  - 参照が実在のファイルを指すか検証

### モジュール 08-checkpoints: 全面深掘り（311 → 800 行超）

- [ ] **T-009** — モジュール導入を実例シナリオで書き直す
  - 例: 「リファクタで 3 つ壊れた。チェックポイントが救う方法は…」

- [ ] **T-010** — チェックポイント戦略の決定ツリーを追加（Mermaid フローチャート）
  - 「X したい」 → 矢印を追う → チェックポイントパターンに辿り着く

- [ ] **T-011** — 3 つのワークフローテンプレートを追加
  - 安全なリファクタリングワークフロー
  - チェックポイントを使った A/B テスト
  - 災害復旧ワークフロー

- [ ] **T-012** — プランニングモードとの統合セクションを追加

- [ ] **T-013** — チェックポイントライフサイクルの Mermaid 図を追加

- [ ] **T-014** — 検証可能なコマンドを伴う「Try It Now」ブロックを 2 つ以上追加
  - 各ブロック: 文脈の文章、実行コマンド、期待される出力の説明

- [ ] **T-015** — 「Patterns & Recipes」セクションに 2 つ以上の名前付きパターンを追加
  - 各パターン: 課題、解決コード、適用すべき場面、適用すべきでない場面

### モジュール 06-hooks: 全面深掘り

- [ ] **T-016** — モジュール導入を実例シナリオで書き直す
  - 例: 「CI が 20 分後に lint エラーを検出した。フックなら 2 秒で…」

- [ ] **T-017** — フックの決定ツリーを追加（Mermaid フローチャート）
  - 「Y が起きたとき X をトリガーしたい」 → 矢印を追う → フックタイプに辿り着く

- [ ] **T-018** — 完全なスクリプト付きの 5 つのフック例を追加（各 80 行程度）
  1. 書き込み時の自動フォーマット（prettier／black）
  2. コミット時のセキュリティスキャン（trufflehog／bandit）
  3. ファイル変更時のテスト実行
  4. セッション終了時の通知
  5. プロンプトの検証（危険なパターンをブロック）

- [ ] **T-019** — 検証可能なコマンドを伴う「Try It Now」ブロックを 2 つ以上追加

- [ ] **T-020** — 「Patterns & Recipes」セクションに 2 つ以上の名前付きパターンを追加

**M1 チェックポイント: インフラ稼働、最も弱い 2 モジュールが完全刷新**
