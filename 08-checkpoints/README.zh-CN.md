# 检查点和回退 (Checkpoints and Rewind)

检查点允许你保存对话状态并回退到 Claude Code 会话中的先前时间点。这对于探索不同方法、从错误中恢复或比较替代解决方案非常有价值。

## 概述

检查点允许你保存对话状态并回退到先前时间点，从而安全地试验和探索多种方法。它们是会话状态的快照，包括：
- 已交换的所有消息
- 所做的文件修改
- 工具使用历史
- 会话上下文

检查点在探索不同方法、从错误恢复或比较替代解决方案时非常宝贵。

## 关键概念

| 概念 | 描述 |
|---------|-------------|
| **检查点** | 会话状态快照，包含消息、文件和上下文 |
| **回退** | 返回到先前的检查点，丢弃后续更改 |
| **分支点** | 从此处探索多种方法的检查点 |

## 访问检查点

你可以通过以下两种主要方式访问和管理检查点：

### 使用键盘快捷键
按两次 `Esc`（`Esc` + `Esc`）打开检查点界面并浏览已保存的检查点。

### 使用斜杠命令
使用 `/rewind` 命令（别名：`/checkpoint`）快速访问：

```bash
# 打开回退界面
/rewind

# 或使用别名
/checkpoint
```

## 回退选项

当你回退时，系统会提供五个选项的菜单：

1. **恢复代码和对话** -- 将文件和消息都回退到该检查点
2. **恢复对话** -- 仅回退消息，保持当前代码不变
3. **恢复代码** -- 仅回退文件更改，保持完整对话历史
4. **从此处摘要** -- 将此点之后的对话压缩为 AI 生成的摘要，而非丢弃。原始消息在转录中保留。你可以可选地提供指令以聚焦摘要的特定主题。
5. **算了** -- 取消并返回到当前状态

## 自动检查点

Claude Code 自动为你创建检查点：

- **每个用户提示** - 每次用户输入都会创建新检查点
- **持久的** - 检查点在会话间持久化
- **自动清理** - 检查点在 30 天后自动清理

这意味着你始终可以回退到对话中的任何先前时间点，从几分钟前到几天前。

## 使用场景

| 场景 | 工作流 |
|----------|----------|
| **探索方法** | 保存 → 尝试 A → 保存 → 回退 → 尝试 B → 比较 |
| **安全重构** | 保存 → 重构 → 测试 → 如果失败：回退 |
| **A/B 测试** | 保存 → 设计 A → 保存 → 回退 → 设计 B → 比较 |
| **错误恢复** | 发现问题 → 回退到最后良好状态 |

## 使用检查点

### 查看和回退

按两次 `Esc` 或使用 `/rewind` 打开检查点浏览器。你将看到带有时间戳的所有可用检查点列表。选择任何检查点进行回退。

### 检查点详情

每个检查点显示：
- 创建时间戳
- 已修改的文件
- 对话中的消息数量
- 使用的工具

## 实际示例

### 示例 1：探索不同方法

```
User: 为 API 添加缓存层

Claude: 我将为你的 API 端点添加 Redis 缓存...
[在检查点 A 进行更改]

User: 实际上，让我们试试内存缓存

Claude: 我将回退以探索不同的方法...
[用户按 Esc+Esc 并回退到检查点 A]
[在检查点 B 实现内存缓存]

User: 现在我可以比较两种方法
```

### 示例 2：从错误中恢复

```
User: 重构认证模块以使用 JWT

Claude: 我将重构认证模块...
[进行大量更改]

User: 等等，那破坏了 OAuth 集成。让我们回去。

Claude: 我将帮你回退到重构之前的状态...
[用户按 Esc+Esc 并选择重构前的检查点]

User: 让我们这次尝试更保守的方法
```

### 示例 3：安全实验

```
User: 让我们尝试用函数式风格重写
[在实验前创建检查点]

Claude: [进行实验性更改]

User: 测试失败了。让我们回退。
[用户按 Esc+Esc 并回退到检查点]

Claude: 我已回退更改。让我们尝试不同的方法。
```

### 示例 4：分支方法

```
User: 我想比较两种数据库设计
[记下检查点 - 称为"开始"]

Claude: 我将创建第一种设计...
[实现方案 A]

User: 现在让我回去试试第二种方法
[用户按 Esc+Esc 并回退到"开始"]

Claude: 现在我将实现方案 B...
[实现方案 B]

User: 太好了！现在我有两种方案可以选择
```

## 检查点保留

Claude Code 自动管理你的检查点：

- 检查点由每个用户提示自动创建
- 旧检查点保留最多 30 天
- 检查点自动清理，以防止无限的存储增长

## 工作流模式

### 探索的分支策略

当探索多种方法时：

```
1. 从初始实现开始 → 检查点 A
2. 尝试方法 1 → 检查点 B
3. 回退到检查点 A
4. 尝试方法 2 → 检查点 C
5. 比较 B 和 C 的结果
6. 选择最佳方法并继续
```

### 安全重构模式

当进行大量更改时：

```
1. 当前状态 → 检查点（自动）
2. 开始重构
3. 运行测试
4. 如果测试通过 → 继续工作
5. 如果测试失败 → 回退并尝试不同的方法
```

## 最佳实践

由于检查点自动创建，你可以专注于工作而无需担心手动保存状态。但是，请记住以下实践：

### 有效使用检查点

✅ **应该做的：**
- 回退前查看可用检查点
- 当你想探索不同方向时使用回退
- 保留检查点以比较不同方法
- 了解每个回退选项的作用（恢复代码和对话、恢复对话、恢复代码或摘要）

❌ **不要做的：**
- 仅依赖检查点进行代码保存
- 期望检查点跟踪外部文件系统更改
- 使用检查点替代 git 提交

## 配置

你可以在设置中切换自动检查点：

```json
{
  "autoCheckpoint": true
}
```

- `autoCheckpoint`：启用或禁用在每个用户提示时自动创建检查点（默认：`true`）

## 限制

检查点有以下限制：

- **不跟踪 Bash 命令更改** - 如 `rm`、`mv`、`cp` 等文件系统操作不会被检查点捕获
- **不跟踪外部更改** - 在 Claude Code 之外进行的更改（在你的编辑器、终端等中）不会被捕获
- **不能替代版本控制** - 使用 git 进行持久、可审计的代码库更改

## 故障排除

### 缺少检查点

**问题**：找不到预期的检查点

**解决方案**：
- 检查检查点是否被清除
- 验证设置中是否启用了 `autoCheckpoint`
- 检查磁盘空间

### 回退失败

**问题**：无法回退到检查点

**解决方案**：
- 确保没有未提交的更改冲突
- 检查检查点是否已损坏
- 尝试回退到不同的检查点

## 与 Git 的集成

检查点补充（但不替代）git：

| 特性 | Git | 检查点 |
|---------|-----|-------------|
| 范围 | 文件系统 | 对话 + 文件 |
| 持久性 | 永久 | 基于会话 |
| 粒度 | 提交 | 任何时间点 |
| 速度 | 较慢 | 即时 |
| 共享 | 是 | 有限 |

结合使用：
1. 使用检查点进行快速实验
2. 使用 git 提交进行最终确定的更改
3. 在 git 操作前创建检查点
4. 将成功的检查点状态提交到 git

## 快速开始指南

### 基本工作流

1. **正常工作** - Claude Code 自动创建检查点
2. **想回去？** - 按两次 `Esc` 或使用 `/rewind`
3. **选择检查点** - 从列表中选择以回退
4. **选择恢复内容** - 选择恢复代码和对话、恢复对话、恢复代码、从此处摘要或取消
5. **继续工作** - 你已回到那个时间点

### 键盘快捷键

- **`Esc` + `Esc`** - 打开检查点浏览器
- **`/rewind`** - 访问检查点的替代方式
- **`/checkpoint`** - `/rewind` 的别名

## 知道何时回退：上下文监控

检查点让你回去——但你如何知道*何时*应该回退？随着对话增长，Claude 的上下文窗口会填满，模型质量会悄然下降。你可能在没有意识到的情况下从一个半盲的模型输出代码。

**[cc-context-stats](https://github.com/luongnv89/cc-context-stats)** 通过在你的 Claude Code 状态栏添加实时**上下文区域**来解决这个问题。它跟踪你在上下文窗口中的位置——从**Plan**（绿色，安全规划和编码）到**Code**（黄色，避免开始新计划）再到**Dump**（橙色，完成工作并回退）。当你看到区域变化时，你知道是时候创建检查点并重新开始，而不是继续输出降级的结果。

## 摘要

检查点是 Claude Code 中的自动功能，让你安全地探索不同方法而无需担心丢失工作。每个用户提示都会自动创建新检查点，因此你可以回退到会话中的任何先前时间点。

主要优点：
- 大胆尝试多种方法
- 快速从错误中恢复
- 并排比较不同解决方案
- 与版本控制系统安全集成

记住：检查点不是 git 的替代品。使用检查点进行快速实验，使用 git 进行持久代码更改。
