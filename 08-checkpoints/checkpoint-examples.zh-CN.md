# 检查点（Checkpoint）示例

在 Claude Code 中有效使用检查点的真实示例。

注意：检查点会在每次用户提示时自动创建。你无需手动保存它们。要回退，按两次 `Esc`（`Esc+Esc`）或使用 `/rewind` 打开检查点浏览器。

## 示例 1：数据库迁移

### 场景
你需要从 MySQL 迁移到 PostgreSQL，但想探索直接迁移和渐进式双写两种方式。

### 工作流

```
用户：让我们实现从 MySQL 到 PostgreSQL 的直接迁移
      [检查点 A 自动创建]

Claude：我将实现直接迁移...
[更改数据库驱动、更新查询、修改模式]

用户：运行测试
      [检查点 B 自动创建]

Claude：运行测试...
❌ 15 个测试失败 - 连接池问题

用户：这种方法有问题。让我们改用双写模式。
[用户按 Esc+Esc，选择检查点 A，选择"恢复代码和对话"]

用户：现在实现一个双写模式，同时写入两个数据库

Claude：我将实现双写迁移...
[实现双写层、渐进式迁移策略]

用户：运行测试

Claude：运行测试...
✅ 所有测试通过

用户：双写方式显然更安全。让我们提交它。
```

## 示例 8：使用检查点的 Summarize 功能

### 场景
经过长时间的调试会话后，你想压缩对话同时保留上下文。

### 工作流

```
用户：[经过 20+ 轮调试和探索的消息]

[用户按 Esc+Esc，选择早期检查点，选择"从此处开始总结"]
[可选地提供指示："重点关注我们尝试了什么以及什么有效"]

Claude：[生成从该点开始的对话摘要]
[原始消息保留在转录中]
[摘要替换了可见的对话，减少了上下文窗口占用]

用户：现在让我们继续有效的方法。
```

## 关键要点

1. **检查点是自动的**：每次用户提示都会创建检查点——无需手动保存
2. **使用 Esc+Esc 或 /rewind**：这是访问检查点浏览器的两种方式
3. **选择正确的恢复选项**：根据需求恢复代码、对话、两者都恢复，或进行总结
4. **不要害怕实验**：检查点让尝试大幅更改变得安全
5. **与 git 结合使用**：用检查点进行探索，用 git 保存最终工作
6. **总结长对话**：使用"从此处开始总结"来保持对话可控
