name: blog-draft
description: Draft a blog post from ideas and resources. Use when users want to write a blog post, create content from research, or draft articles. Guides through research, brainstorming, outlining, and iterative drafting with version control.
---

## 用户输入

```text
$ARGUMENTS
```

在继续之前，**必须**先考虑用户输入（如果不为空）。用户应提供：
- **想法/主题**：博文的主要概念或主题
- **资源**：用于研究的 URL、文件或参考资料（可选但推荐）
- **目标受众**：博文面向的人群（可选）
- **语气/风格**：正式、随意、技术性等等（可选）

**重要提示**：如果用户要求更新**已有博文**，跳过第 0-8 步，直接从**第 9 步**开始。先阅读现有的草稿文件，然后进入迭代流程。

## 执行流程

以下列步骤按顺序执行。**不要跳过步骤，也未获得用户批准不能继续。**

### 第 0 步：创建项目文件夹

1. 生成文件夹名称，格式：`YYYY-MM-DD-short-topic-name`
   - 使用今天的日期
   - 从主题中创建简短的、对 URL 友好的标识符（小写、连字符，最多 5 个单词）

2. 创建文件夹结构：
   ```
   blog-posts/
   └── YYYY-MM-DD-short-topic-name/
       └── resources/
   ```

3. 在继续之前向用户确认文件夹创建。

### 第 1 步：研究与资源收集

1. 在博文目录中创建 `resources/` 子文件夹

2. 对于每个提供的资源：
   - **URL**：抓取并将关键信息保存为 `resources/` 中的 Markdown 文件
   - **文件**：阅读并在 `resources/` 中总结
   - **主题**：使用网络搜索收集最新信息

3. 对于每个资源，在 `resources/` 中创建一个摘要文件：
   - `resources/source-1-[short-name].md`
   - `resources/source-2-[short-name].md`
   - 以此类推

4. 每个摘要应包括：
   ```markdown
   # 来源：[标题/URL]

   ## 关键要点
   - 要点 1
   - 要点 2

   ## 相关引用/数据
   - 引用或统计数据 1
   - 引用或统计数据 2

   ## 与主题的关联
   简要说明相关性
   ```

5. 向用户展示研究摘要。

### 第 2 步：头脑风暴与澄清

1. 基于想法和研究的资源，呈现：
   - 从研究中识别的**主要主题**
   - 博文的**潜在角度**
   - 应涵盖的**关键点**
   - 需要澄清的**信息空白**

2. 提出澄清问题：
   - 你希望读者获得的核心收获是什么？
   - 研究中是否有你想特别强调的观点？
   - 目标长度是多少？（短文：500-800 字，中等：1000-1500，长文：2000 字以上）
   - 有哪些观点需要排除？

3. **在继续之前等待用户回复。**

### 第 3 步：提议大纲

1. 创建结构化的大纲，包括：

   ```markdown
   # 博文大纲：[标题]

   ## 元信息
   - **目标受众**：[谁]
   - **语气**：[风格]
   - **目标长度**：[字数]
   - **核心收获**：[关键信息]

   ## 建议结构

   ### 钩子/引言
   - 开场钩子的想法
   - 背景设定
   - 论点陈述

   ### 第 1 部分：[标题]
   - 关键点 A
   - 关键点 B
   - 来自 [来源] 的支撑证据

   ### 第 2 部分：[标题]
   - 关键点 A
   - 关键点 B

   [为所有部分继续...]

   ### 结论
   - 关键点总结
   - 行动号召或最终想法

   ## 引用来源
   - 来源 1
   - 来源 2
   ```

2. 向用户展示大纲并**征求批准或修改意见**。

### 第 4 步：保存已批准的大纲

1. 用户批准大纲后，将其保存到博文文件夹中的 `OUTLINE.md`。

2. 确认大纲已保存。

### 第 5 步：提交大纲（如果在 Git 仓库中）

1. 检查当前目录是否是 Git 仓库。

2. 如果是：
   - 暂存新文件：博文文件夹、资源和 OUTLINE.md
   - 创建提交，提交信息：`docs: Add outline for blog post - [topic-name]`
   - 推送到远程仓库

3. 如果不是 Git 仓库，跳过此步骤并告知用户。

### 第 6 步：撰写草稿

1. 基于已批准的大纲，撰写完整的博文草稿。

2. 严格按照 OUTLINE.md 中的结构进行。

3. 包括：
   - 引人入胜的引言（带钩子）
   - 清晰的小节标题
   - 研究中获得的支撑证据和示例
   - 各部分之间的平滑过渡
   - 有力的结论（带核心收获）
   - **引用**：所有比较、统计数据、数据点和事实声明**必须**注明原始来源

4. 将草稿保存为博文文件夹中的 `draft-v0.1.md`。

5. 格式：
   ```markdown
   # [博文标题]

   *[可选：副标题或标语]*

   [完整内容，附带行内引用...]

   ---

    ## 参考文献
   - [1] 来源 1 标题 - URL 或引用
   - [2] 来源 2 标题 - URL 或引用
   - [3] 来源 3 标题 - URL 或引用
   ```

6. **引用要求**：
   - 每个数据点、统计数据或比较都必须有行内引用
   - 使用编号引用 [1]、[2] 等，或命名引用 [来源名称]
   - 将引用链接到末尾的参考文献部分
   - 示例："研究表明 65% 的开发者更喜欢 TypeScript [1]"
   - 示例："React 在渲染速度上比 Vue 快 20% [React Benchmarks 2024]"

### 第 7 步：提交草稿（如果在 Git 仓库中）

1. 检查是否在 Git 仓库中。

2. 如果是：
   - 暂存草稿文件
   - 创建提交，提交信息：`docs: Add draft v0.1 for blog post - [topic-name]`
   - 推送到远程仓库

3. 如果不是 Git 仓库，跳过并告知用户。

### 第 8 步：提交草稿供审阅

1. 向用户展示草稿内容。

2. 征求反馈：
   - 整体印象如何？
   - 哪些部分需要扩展或缩减？
   - 语气需要调整吗？
   - 是否有缺失的信息？
   - 是否有具体的修改或重写要求？

3. **等待用户回复。**

### 第 9 步：迭代或定稿

**如果用户要求修改：**
1. 记录所有请求的修改
2. 回到第 6 步，进行以下调整：
   - 递增版本号（v0.2、v0.3 等）
   - 整合所有反馈
   - 保存为 `draft-v[X.Y].md`
   - 重复第 7-8 步

**如果用户批准：**
1. 确认最终草稿版本
2. 如果用户要求，可选重命名为 `final.md`
3. 总结博文创建过程：
   - 创建的版本总数
   - 版本之间的关键变化
   - 最终字数
   - 创建的文件

## 版本追踪

所有草稿都采用递增版本号保留：
- `draft-v0.1.md` - 初始草稿
- `draft-v0.2.md` - 第一轮反馈后
- `draft-v0.3.md` - 第二轮反馈后
- 以此类推

这样可以追踪博文的演变过程，并在需要时回退。

## 输出文件结构

```
blog-posts/
└── YYYY-MM-DD-topic-name/
    ├── resources/
    │   ├── source-1-name.md
    │   ├── source-2-name.md
    │   └── ...
    ├── OUTLINE.md
    ├── draft-v0.1.md
    ├── draft-v0.2.md（如有迭代）
    └── draft-v0.3.md（如有更多迭代）
```

## 质量提示

- **钩子**：以一个问题、令人惊讶的事实或可共鸣的场景开头
- **流畅**：每个段落都应连接到下一个
- **证据**：用研究数据支撑声明
- **引用**：始终为以下内容注明来源：
  - 所有统计数据和数据点（例如，"根据 [来源]，75% 的..."）
  - 产品、服务或方法之间的比较（例如，"X 的性能是 Y 的 2 倍 [来源]"）
  - 关于市场趋势、研究发现或基准测试的事实声明
  - 使用行内引用格式：[来源名称] 或 [作者, 年份]
- **语气**：全文保持一致的语调
- **长度**：尊重目标字数
- **可读性**：使用简短段落，适当使用列表
- **行动号召**：以清晰的行动号召或发人深省的问题结尾

## 注意事项

- 始终在规定的检查点等待用户批准
- 保留所有草稿版本以便查阅历史
- 当提供 URL 时，使用网络搜索获取最新信息
- 如果资源不足，向用户索取更多或建议补充研究
- 根据目标受众调整语气（技术性、通识、商业等）
