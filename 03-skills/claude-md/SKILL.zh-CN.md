## 用户输入

```text
$ARGUMENTS
```

在继续之前，**必须**先考虑用户输入（如果不为空）。用户可以指定：
- `create` - 从头创建新的 CLAUDE.md
- `update` - 改进现有的 CLAUDE.md
- `audit` - 分析并报告当前 CLAUDE.md 的质量
- 创建/更新的特定路径（例如 `src/api/CLAUDE.md` 用于目录特定指令）

## 核心原则

**LLM 是无状态的**：CLAUDE.md 是唯一在每次对话中自动包含的文件。它作为 AI 代理进入代码库的主要入门文档。

### 黄金法则

1. **少即是多**：前沿 LLM 可以遵循大约 150-200 条指令。Claude Code 的系统提示已经使用了大约 50 条。保持你的 CLAUDE.md 专注简洁。

2. **通用适用性**：只包含与**每次**会话都相关的信息。特定任务的说明应放在单独的文件中。

3. **不要把 Claude 当作 Linter**：风格指南会膨胀上下文并降低指令遵循能力。使用确定性工具（prettier、eslint 等）代替。

4. **不要自动生成**：CLAUDE.md 是 AI 工具最高效力的切入点。需要仔细考虑后手动编写。

## 执行流程

### 1. 项目分析

首先，分析当前项目状态：

1. 检查现有的 CLAUDE.md 文件：
   - 根级别：`./CLAUDE.md` 或 `.claude/CLAUDE.md`
   - 目录特定：`**/CLAUDE.md`
   - 全局用户配置：`~/.claude/CLAUDE.md`

2. 识别项目结构：
   - 技术栈（语言、框架）
   - 项目类型（单体仓库、单个应用、库）
   - 开发工具（包管理器、构建系统、测试运行器）

3. 查看现有文档：
   - README.md
   - CONTRIBUTING.md
   - package.json、pyproject.toml、Cargo.toml 等

### 2. 内容策略（WHAT、WHY、HOW）

围绕三个维度构建 CLAUDE.md 的结构：

#### WHAT - 技术与结构
- 技术栈概述
- 项目组织（对单体仓库尤为重要）
- 关键目录及其用途

#### WHY - 目的与上下文
- 项目的功能
- 某些架构决策的原因
- 每个主要组件的职责

#### HOW - 工作流与约定
- 开发工作流（bun 还是 node、pip 还是 uv 等）
- 测试流程和命令
- 验证和构建方法
- 关键的"陷阱"或不明显的要求

### 3. 渐进式披露策略

对于大型项目，建议创建 `agent_docs/` 文件夹：

```
agent_docs/
  |- building_the_project.md
  |- running_tests.md
  |- code_conventions.md
  |- architecture_decisions.md
```

在 CLAUDE.md 中引用这些文件，指令如下：
```markdown
有关详细的构建说明，请参阅 `agent_docs/building_the_project.md`
```

**重要提示**：使用 `file:line` 引用而不是代码片段，以避免过时上下文。

### 4. 质量约束

在创建或更新 CLAUDE.md 时：

1. **目标长度**：不超过 300 行（理想情况下不超过 100 行）
2. **无样式规则**：删除任何格式化/格式化指令
3. **无特定任务指令**：移至单独的文件
4. **无代码片段**：使用文件引用代替
5. **无冗余信息**：不要重复 package.json 或 README 中的内容

### 5. 基本部分

结构良好的 CLAUDE.md 应包括：

```markdown
# 项目名称

简短的一行描述。

## 技术栈
- 主要语言及版本
- 关键框架/库
- 数据库/存储（如果有）

## 项目结构
[仅适用于单体仓库或复杂结构]
- `apps/` - 应用入口点
- `packages/` - 共享库

## 开发命令
- 安装：`命令`
- 测试：`命令`
- 构建：`命令`

## 关键约定
[仅限不明显的、影响重大的约定]
- 约定 1 及简要说明
- 约定 2 及简要说明

## 已知问题 / 陷阱
[经常让开发者困惑的事情]
- 问题 1
- 问题 2
```

### 6. 应避免的反模式

**不要包括：**
- 代码风格指南（使用 linter）
- 如何使用 Claude 的说明
- 对明显模式的冗长解释
- 复制粘贴的代码示例
- 通用的最佳实践（"编写干净的代码"）
- 特定任务的指令
- 自动生成的内容
- 大量的 TODO 列表

### 7. 验证清单

在定稿之前，确认：

- [ ] 不超过 300 行（最好不到 100 行）
- [ ] 每行内容都适用于**所有**会话
- [ ] 没有样式/格式规则
- [ ] 没有代码片段（使用文件引用）
- [ ] 命令经过验证可以正常工作
- [ ] 对复杂项目使用了渐进式披露
- [ ] 关键的陷阱事项已记录
- [ ] 没有与 README.md 重复

## 输出格式

### 用于 `create` 或默认情况：

1. 分析项目
2. 起草遵循上述结构的 CLAUDE.md
3. 展示草稿供审阅
4. 获得批准后写入适当的位置

### 用于 `update`：

1. 阅读现有的 CLAUDE.md
2. 根据最佳实践进行审计
3. 识别：
   - 需要删除的内容（样式规则、代码片段、特定任务内容）
   - 需要精简的内容
   - 缺失的关键信息
4. 展示修改建议供审阅
5. 获得批准后应用更改

### 用于 `audit`：

1. 阅读现有的 CLAUDE.md
2. 生成报告，包括：
   - 当前行数与目标比较
   - 通用适用内容的百分比
   - 发现的反模式列表
   - 改进建议
3. **不要**修改文件，仅报告

## AGENTS.md 处理

如果用户请求创建/更新 AGENTS.md：

AGENTS.md 用于定义特定的代理行为。与 CLAUDE.md（用于项目上下文）不同，AGENTS.md 定义：
- 自定义代理角色和功能
- 特定代理的指令和约束
- 多代理场景的工作流定义

应用类似原则：
- 保持专注简洁
- 使用渐进式披露
- 引用外部文档而不是嵌入内容

## 注意事项

- 始终在包含命令之前验证它们可以正常工作
- 如果不确定，就别写——少即是多
- 系统提醒告诉 Claude 那 CLAUDE.md"可能是或可能不相关"——噪音越多，越容易被忽略
- 单体仓库从清晰的 WHAT/WHY/HOW 结构中获益最大
- 目录特定的 CLAUDE.md 文件应更加专注
