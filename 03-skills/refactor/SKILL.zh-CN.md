# 代码重构技能

一种基于 Martin Fowler 的《重构：改善既有代码的设计》（第 2 版）的系统化重构方法。本技能强调通过测试支撑的安全、增量式更改。

> "重构是这样一个过程：在不改变代码外部行为的前提下，改善其内部结构。" — Martin Fowler

## 核心原则

1. **行为保持**：外部行为必须保持不变
2. **小步前进**：进行微小的、可测试的更改
3. **测试驱动**：测试是安全网
4. **持续性**：重构是持续性的，而非一次性事件
5. **协作性**：每个阶段需要用户确认

## 工作流概述

```
阶段 1：研究与分析
    ↓
阶段 2：测试覆盖率评估
    ↓
阶段 3：代码异味识别
    ↓
阶段 4：制定重构计划
    ↓
阶段 5：增量式实施
    ↓
阶段 6：审查与迭代
```

---

## 阶段 1：研究与分析

### 目标
- 理解代码库结构和用途
- 确定重构的范围
- 收集关于业务需求的信息

### 需要向用户确认的问题
在开始之前，需要明确：

1. **范围**：哪些文件/模块/函数需要重构？
2. **目标**：你试图解决什么问题？（可读性、性能、可维护性）
3. **约束**：有哪些不应更改的区域？
4. **时间压力**：这是否会阻塞其他工作？
5. **测试状态**：是否有测试？它们是否通过？

### 行动
- [ ] 阅读并理解目标代码
- [ ] 识别依赖关系和集成
- [ ] 记录当前架构
- [ ] 记录任何现有的技术债务标记（TODO、FIXME）

### 输出
向用户展示发现：
- 代码结构摘要
- 识别出的问题区域
- 初步建议
- **请求批准以继续**

## 阶段 2：测试覆盖率评估

### 为什么测试很重要
> "没有测试的重构就像是不系安全带开车。" — Martin Fowler

测试是安全重构的**关键前提**。没有它们，就有可能引入缺陷。

### 评估步骤

1. **检查现有测试**
   ```bash
   # 查找测试文件
   find . -name "*test*" -o -name "*spec*" | head -20
   ```

2. **运行现有测试**
   ```bash
   # JavaScript/TypeScript
   npm test

   # Python
   pytest -v

   # Java
   mvn test
   ```

3. **检查覆盖率（如果可用）**
   ```bash
   # JavaScript
   npm run test:coverage

   # Python
   pytest --cov=.
   ```

### 决策点：询问用户

**如果测试存在并通过：**
- 进入阶段 3

**如果测试缺失或不完整：**
提供选项：
1. 先编写测试（推荐）
2. 在重构过程中增量添加测试
3. 无测试继续（有风险 - 需要用户确认）

**如果测试失败：**
- 停止。在重构前先修复失败的测试
- 询问用户：我们应该先修复测试吗？

### 测试编写指南（如需要）

对于每个正在重构的函数，确保测试覆盖：
- 正常路径（正常操作）
- 边界情况（空输入、null、边界值）
- 错误场景（无效输入、异常）

使用"红-绿-重构"循环：
1. 编写失败的测试（红）
2. 使其通过（绿）
3. 重构

## 阶段 3：代码异味识别

### 什么是代码异味？
代码中深层问题的症状。它们不是缺陷，但是表明代码可能可以改进的信号。

### 要检查的常见代码异味

完整的目录请参考 [references/code-smells.zh-CN.md](references/code-smells.zh-CN.md)。

#### 快速参考

| 异味 | 表现 | 影响 |
|-------|-------|--------|
| **长方法** | 方法超过 30-50 行 | 难以理解、测试和维护 |
| **重复代码** | 相同逻辑出现在多个地方 | Bug 修复需要在多处修改 |
| **大类** | 承担太多职责的类 | 违反单一职责原则 |
| **特性依恋** | 方法过多使用另一个类数据 | 封装不良 |
| **基本类型偏执** | 过度使用基本类型而非对象 | 缺少领域概念 |
| **长参数列表** | 方法有 4 个以上参数 | 难以正确调用 |
| **数据泥团** | 相同数据项总是同时出现 | 缺少抽象 |
| **条件表达式** | 复杂的 switch/if-else 链 | 难以扩展 |
| **过度泛化** | "以防万一"的代码 | 不必要的复杂性 |
| **死代码** | 未使用的代码 | 困惑和维护负担 |

### 分析步骤

1. **自动分析**（如有可用脚本）
   ```bash
   python scripts/detect-smells.py <file>
   ```

2. **手动检查**
   - 系统性地遍历代码
   - 记录每个异味的位置和严重性
   - 按影响分类（关键/高/中/低）

3. **优先级排序**
   关注以下异味：
   - 阻塞当前开发的
   - 导致 Bug 或困惑的
   - 影响最常变更的代码路径的

### 输出：异味报告

向用户展示：
- 已识别的异味列表及其位置
- 每种异味的严重性评估
- 建议的优先级顺序
- **请求确认优先级**

## 阶段 4：制定重构计划

### 选择重构

对于每种异味，从目录中选择合适的重构。

完整的列表请参考 [references/refactoring-catalog.zh-CN.md](references/refactoring-catalog.zh-CN.md)。

#### 异味到重构的映射

| 代码异味 | 推荐的重构方法 |
|------------|---------------------------|
| 长方法 | 提取函数、用查询取代临时变量 |
| 重复代码 | 提取函数、方法上移、形成模板方法 |
| 大类 | 提取类、提取子类 |
| 特性依恋 | 方法移动、字段移动 |
| 基本类型偏执 | 以对象取代基本类型、以类取代类型码 |
| 长参数列表 | 引入参数对象、保持对象完整 |
| 数据泥团 | 提取类、引入参数对象 |
| 条件表达式 | 以多态取而代之 |
| 过度泛化 | 折叠继承体系、内嵌类、移除死代码 |
| 死代码 | 移除死代码 |

### 计划结构

使用模板 [templates/refactoring-plan.zh-CN.md](templates/refactoring-plan.zh-CN.md)。

对于每个重构：
1. **目标**：哪些代码将发生变化
2. **异味**：要解决什么问题
3. **重构**：应用哪种技术
4. **步骤**：详细的微步骤
5. **风险**：可能出什么问题
6. **回滚**：如何在需要时撤销

### 分阶段方法

**关键**：分阶段逐步引入重构。

**阶段 A：快速取胜**（低风险，高价值）
- 为变量重命名以提高清晰度
- 提取明显的重复代码
- 移除死代码

**阶段 B：结构性改进**（中等风险）
- 从长函数中提取方法
- 引入参数对象
- 将方法移动到适当的类

**阶段 C：架构变更**（较高风险）
- 以多态取代条件表达式
- 提取类
- 引入设计模式

### 决策点：向用户展示计划

在实施之前：
- 展示完整的重构计划
- 解释每个阶段及其风险
- 获取每个阶段的明确批准
- **询问**："我应该开始阶段 A 吗？"

## 阶段 5：增量式实施

### 黄金法则
> "更改 -> 测试 -> 通过？-> 提交 -> 下一步"

### 实施节奏

对于每个重构步骤：

1. **预检查**
   - 测试已通过的（绿）
   - 代码可编译

2. **进行一次小的更改**
   - 遵循目录中的操作方法
   - 保持更改最小化

3. **验证**
   - 立即运行测试
   - 检查编译错误

4. **如果测试通过（绿）**
   - 用描述性的消息提交
   - 进入下一步

5. **如果测试失败（红）**
   - 立即停止
   - 撤销更改
   - 分析出了什么问题
   - 如果不清楚，向用户询问

### 提交策略

每次提交应：
- **原子化**：一次逻辑更改
- **可回滚**：易于回退
- **有描述**：清晰的提交信息

提交信息示例：
```
refactor: 从 processOrder() 中提取 calculateTotal()
refactor: 将 'x' 重命名为 'customerCount' 以提高清晰度
refactor: 移除未使用的 validateOldFormat() 方法
```

### 进度报告

每个子阶段完成后，向用户报告：
- 所做的更改
- 测试是否还通过？
- 遇到了什么问题
- **询问**："继续下一批吗？"

## 阶段 6：审查与迭代

### 重构后检查清单

- [ ] 所有测试通过
- [ ] 没有新的警告/错误
- [ ] 代码编译成功
- [ ] 行为不变（手动验证）
- [ ] 文档已更新（如需要）
- [ ] 提交历史清晰

### 指标对比

重构前后运行复杂度分析：
```bash
python scripts/analyze-complexity.py <file>
```

展示改进：
- 代码行数变化
- 圈复杂度变化
- 可维护性指数变化

### 用户审查

展示最终结果：
- 所有变更摘要
- 前后代码对比
- 指标改进
- 剩余的技术债务
- **询问**："你对这些更改满意吗？"

### 下一步

与用户讨论：
- 是否要处理其他异味？
- 是安排跟进的重构？
- 是否在其他地方应用类似的更改？

## 重要指南

### 何时暂停并询问

遇到以下情况时始终暂停并咨询用户：
- 不确定业务逻辑
- 更改可能影响外部 API
- 测试覆盖不够
- 需要做出重大架构决策
- 风险等级上升
- 遇到未预料到的复杂性

### 安全规则

1. **没有测试绝不重构**（除非用户明确确认风险）
2. **绝不大刀阔斧的更改** — 分解为微小步骤
3. **每次更改后绝不跳过测试**
4. **测试失败时绝不继续** — 先修复或回滚
5. **绝不假设** - 有疑问就问

### 不要做的事情

- 不要将重构与功能添加混合
- 不要在生产环境紧急情况下重构
- 不要重构你不理解的代码
- 不要过度工程化 - 保持简单
- 不要一次性重构所有东西

## 快速入门示例

### 场景：包含重复的长方法

**重构前：**
```javascript
function processOrder(order) {
  // 150 行代码，包含：
  // - 重复的验证逻辑
  // - 内联计算
  // - 混合的职责
}
```

**重构步骤：**

1. **确保存在测试** - 用于 processOrder()
2. **提取** 验证逻辑到 validateOrder()
3. **测试** - 应该通过
4. **提取** 计算逻辑到 calculateOrderTotal()
5. **测试** - 应该通过
6. **提取** 通知逻辑到 notifyCustomer()
7. **测试** - 应该通过
8. **审查** - processOrder() 现在编排 3 个清晰的函数

**重构后：**
```javascript
function processOrder(order) {
  validateOrder(order);
  const total = calculateOrderTotal(order);
  notifyCustomer(order, total);
  return { order, total };
}
```
