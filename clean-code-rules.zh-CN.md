# AI 代码生成的整洁代码规则

这些规则指导代码生成，产出可维护、专业质量的代码。

## 有意义的命名

- 使用能揭示意图的名称，解释某事物为何存在
- 避免误导性信息和无意义的区分（如 `data`、`info`、`manager`）
- 使用可发音、可搜索的名称
- 类名：名词（如 `UserAccount`、`PaymentProcessor`）
- 方法名：动词（如 `calculateTotal`、`sendEmail`）
- 避免心智映射和编码（匈牙利表示法、前缀）

## 函数

- 保持函数短小（理想情况 < 20 行）
- 只做一件事 — 单一职责原则
- 每个函数保持一个抽象层级
- 限制参数数量：0-2 个理想，最多 3 个，避免布尔参数
- 无副作用 — 函数应做其名称所描述的事
- 分离命令（改变状态）和查询（返回信息）
- 使用异常而非错误码

## 注释

- 代码应当自解释 — 尽可能避免注释
- 好的注释：法律信息、警告、TODO、公共 API 文档
- 坏的注释：冗余的、误导性的或为解释烂代码的
- 永远不要注释掉代码 — 删除它（版本控制保留历史）
- 如果需要注释，考虑重构代码instead

## 格式化

- 保持文件短小且专注
- 垂直格式化：相关概念靠近放置，空行分隔不同概念
- 水平格式化：限制行长度（80-120 字符）
- 使用一致的缩进和团队风格
- 将相关函数组织在一起

## 对象和数据结构

- 对象：将数据隐藏在抽象后面，通过方法暴露行为
- 数据结构：暴露数据，行为最小化
- 迪米特法则：只与直接朋友交谈，避免 `a.getB().getC().doSomething()`
- 不要盲目地通过 getter/setter 暴露内部结构

## 错误处理

- 使用异常而非返回码或错误标志
- 在可能失败的代码中先写 `try-catch-finally`
- 在异常消息中提供上下文
- 不要返回 `null` — 返回空集合或使用 Optional/Maybe
- 不要将 `null` 作为参数传递

## 类

- 类要小：以职责衡量，而非行数
- 单一职责原则：只有一个改变的理由
- 高内聚：类变量被多个方法使用
- 低耦合：类之间的依赖最小
- 开闭原则：对扩展开放，对修改封闭

## 单元测试

- 快速、独立、可重复、自我验证、及时（F.I.R.S.T.）
- 每个测试一个断言（或一个概念）
- 测试代码质量应等于生产代码质量
- 可读的测试名称，描述正在测试的内容
- Arrange-Act-Assert 模式

## 代码质量原则

- **DRY（不要重复自己）**：无重复
- **YAGNI（你不会需要它）**：不要为假设的未来构建
- **KISS（保持简单）**：避免不必要的复杂性
- **童子军规则**：让代码比你发现时更干净

## 要避免的代码坏味道

- 过长的函数或类
- 重复代码
- 死代码（未使用的变量、函数、参数）
- 特性嫉妒（方法对别的类更感兴趣）
- 不适当的亲密关系（类之间互相了解过多）
- 过长的参数列表
- 基本类型偏执（过度使用基本类型而非小型对象）
- switch/case 语句（考虑多态替代）
- 临时字段（仅有时使用的类变量）

## 并发

- 将并发代码与其他代码分离
- 限制同步/锁定数据的范围
- 使用线程安全的集合
- 保持同步区域小巧
- 了解你的执行模型和原语

## 系统设计

- 分离构建与使用（依赖注入）
- 复杂对象创建使用工厂、构建者模式
- 面向接口编程，而非实现
- 优先使用组合而非继承
- 应用设计模式以简化为目的，而非为了炫技

## 重构

- 持续重构，而非大批量进行
- 重构前后测试均应通过
- 小步前进：一次只做一个变更
- 常见重构：提取方法、重命名、移动、内联

## 文档

- 自解释代码 > 注释 > 外部文档
- 公共 API 需要清晰的文档
- 文档中包含示例
- 文档应靠近代码（理想情况下就在代码中）

---

**核心理念**：代码被阅读的次数是编写次数的 10 倍。优化可读性和可维护性，而非追求巧妙。
